Note: This report documents the same dataset, targets, and features that the live valuation tool (app.py, D2) trains on — Forbes Global 2000, EV/Revenue multiples, sector-relative undervaluation. It independently runs the full required pipeline (proper train/validation/test splits, a four-rung model ladder, leakage-safe feature selection) that the production app’s tuned GradientBoosting models do not themselves perform when they train live at startup. app.py now trains three models on this population — a classifier (undervaluation flag), a regressor for EV/Revenue, and a second regressor for EV/EBITDA — this report formally benchmarks the classifier and the EV/Revenue regressor per the assignment’s one-classification/one-regression requirement; the EV/EBITDA extension is summarized in the Conclusions. See D5_executive_summary.md for the plain-language findings.

Executive Summary

This report builds a machine learning valuation framework using the Forbes Global 2000 as a comparable companies dataset — the same methodology investment banks and private equity firms use to value acquisition targets.

Two ML tasks:

  • Regression: Predict the EV/Revenue ratio — the enterprise-value-to-revenue multiple the market assigns to a company with given fundamentals. EV/Revenue is the standard M&A multiple for companies without reliable EBITDA (high-growth, early-stage, or capital-intensive firms).
  • Classification: Predict whether a company is undervalued (EV/Revenue below its industry median) — a first-pass acquisition screening signal.

A four-rung model ladder (Dummy → Linear → Decision Tree → XGBoost) is trained for each task inside a leakage-safe pipeline that also applies Recursive Feature Elimination (RFE).

Key observations:

  • The core Forbes fields (Sales_B, Profits_B, Assets_B, MarketValue_B, Industry, Country) come directly from the Forbes Global 2000 ranking and are complete for essentially every company.
  • description, rev_growth, ebitda_margin, gross_margin, fcf_margin are enrichment fields pulled from Yahoo Finance by scripts/enrich_forbes_descriptions.py — coverage is partial because not every Forbes company resolves cleanly to a Yahoo Finance ticker by name (private companies, small regional firms, non-English names).
  • net_debt_B defaults to 0.0 for companies without a resolved ticker — a conservative assumption (treated as unlevered) for the EV calculation below.
  • A currency-conversion bug was found and fixed during development. Yahoo Finance reports non-USD-reporting companies’ financial fields (debt, cash, market cap) in their local reporting currency, not USD — the enrichment pipeline was originally reading these raw, unconverted. This produced absurd values for the ~170 Japanese, Korean, and other non-USD companies with resolved tickers (e.g. Toyota’s net_debt_B computed at ~30,000 instead of ~187, off by roughly the JPY/USD exchange rate). The fix converts every dollar-denominated field to USD using the ticker’s reported currency (financialCurrency for financial-statement fields, currency for market cap — these can differ for ADRs) before any further computation. This is exactly the kind of silent data-quality issue that’s easy to miss without spot-checking real companies against known figures, and it directly corrupted the training data (net_debt_B, and therefore ev_B/ev_revenue) for every affected row until caught.

Leakage-Safe Preprocessing Pipeline

Code 
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer

def build_preprocessor():
    num_pipe = Pipeline([
        ("imputer", SimpleImputer(strategy="median")),
        ("scaler",  StandardScaler()),
    ])
    cat_pipe = Pipeline([
        ("ohe", OneHotEncoder(handle_unknown="ignore", sparse_output=False)),
    ])
    return ColumnTransformer([
        ("num", num_pipe, NUMERIC_FEATURES),
        ("cat", cat_pipe, CATEGORICAL_FEATURES),
    ])

Feature Selection — Recursive Feature Elimination (RFE)

After one-hot encoding Industry (27 categories), the pipeline produces roughly 36 transformed columns. To satisfy the assignment’s feature-selection requirement without leaking test information, we wrap RFE as a pipeline step that is refit on every training fold:

Code 
from sklearn.feature_selection import RFE
from sklearn.linear_model import LogisticRegression, Ridge

SELECT_FRAC = 0.5  # keep the top half of transformed features

def make_clf_pipeline(model):
    return Pipeline([
        ("pre",    build_preprocessor()),
        ("select", RFE(LogisticRegression(max_iter=1_000, random_state=42),
                       n_features_to_select=SELECT_FRAC)),
        ("model",  model),
    ])

def make_reg_pipeline(model):
    return Pipeline([
        ("pre",    build_preprocessor()),
        ("select", RFE(Ridge(alpha=1.0), n_features_to_select=SELECT_FRAC)),
        ("model",  model),
    ])

print("RFE wrapped inside the Pipeline for every model in the ladder (Part D).")
print("Because it sits after 'pre' inside one Pipeline object, calling .fit(X_train, y_train)")
print("refits the selector on the training fold only — no test-set information leaks in.")

                val_scores.mean(axis=1) + val_scores.std(axis=1), alpha=0.1, color="#C44E52")
ax.axvline(x=4, ls="--", color="grey", label="Sweet spot (depth=4)")
ax.set_xlabel("Decision Tree max_depth")
ax.set_ylabel("ROC-AUC")
ax.set_title("Validation Curve — Decision Tree overfits beyond depth 4-5")

Classic overfitting: train AUC climbs toward 1.0 as depth increases; val AUC peaks around depth 3-5 and then flattens or degrades. We cap the tree at max_depth=4 in the ladder below. This also informs the XGBoost depth prior.

Business Interpretation — M&A Use Case

How to use these models in practice:

  1. For valuing a private acquisition target: Enter the target’s revenue, profit, assets, margins, and industry (the same inputs app.py collects). The regression model returns the EV/Revenue multiple a comparable Forbes Global 2000 public company with those fundamentals commands. Multiply by target revenue to get an implied enterprise value.

  2. For screening acquisition targets among public companies: The classification model flags companies trading below their industry-median multiple — the starting point for a value-oriented acquisition screen.

  3. What the industry table tells you: Software/IT-services and healthcare-equipment command multiples several times higher than banking or utilities at the same revenue level. Industry is therefore one of the most important inputs to any M&A valuation, which is exactly why RFE keeps most of the industry dummies even after eliminating half the transformed feature space.

What I would NOT trust these models to do:

  • Provide a precise valuation for a specific target — treat the output as a directional anchor, not a binding bid number.
  • Replace industry knowledge — the model averages across Forbes G2000 comps; niche sub-industries (SaaS vs. IT distribution, both labeled “IT Software & Services”) behave very differently, which is exactly what the live app’s description-based TF-IDF matching (D2) is designed to correct for.
  • Generalise to small-cap or private company universes — Forbes G2000 companies are large-cap by definition and their multiples reflect different liquidity and risk premiums.
  • Extrapolate without retraining — rate environment and sector rotation shift multiples significantly over time.

Both XGBoost models outperform their Dummy baselines. The classification task demonstrates that company fundamentals — particularly profit margin, industry, and scale — predict whether a company trades at a discount to industry peers, even after RFE removes half the transformed feature space. The regression task provides a data-driven EV/Revenue estimate that can anchor M&A valuation analysis when a formal comparable-companies table is unavailable.

Production extension — EV/EBITDA. app.py trains a third model beyond what this report formally benchmarks: a second GradientBoosting regressor predicting EV/EBITDA, using the same feature set and log-transform approach as the EV/Revenue regressor above, trained on the ~34% of the population with usable EBITDA coverage (a smaller, noisier subset than the ~1,901-row EV/Revenue population). It’s surfaced in the app as a parallel, explicitly lower-confidence metric alongside the EV/Revenue estimate — not blended into it — since EV/EBITDA is often the more appropriate multiple for capital-intensive businesses where EV/Revenue understates true earning power. This report doesn’t run a full second ladder for it, since the assignment requires one regression target, not two; the EV/Revenue ladder above is the one formally validated with a train/validation/test split and a leaderboard.

Production verdict methodology. For a loaded public company, app.py’s verdict compares the company’s actual EV/Revenue against a blended benchmark — 30% the trained model’s own prediction, 70% the peer median from the comparable-companies table — rather than the model’s raw prediction alone. This weights the verdict toward what similar real companies are actually trading at, with the model term correcting for idiosyncrasies the peer group doesn’t fully capture. For private/manually-entered companies with no market price to compare against, the app deliberately does not show an undervalued/overvalued/fair-value judgment — there’s nothing to compare it to — and instead presents the blended figure as a plain estimated valuation.

Full reproducible code is in this repository. The live Streamlit app (app.py) trains all three tuned GradientBoosting models (classifier, EV/Revenue regressor, EV/EBITDA regressor) on this same population for interactive, per-company valuation queries.
